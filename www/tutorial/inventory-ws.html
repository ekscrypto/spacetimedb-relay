6. Live inventory WebSocket

One multiplexed socket streams personal bags, housing interiors, claim shared storage, and crafts as they change — without polling HTTP or opening dozens of connections. Built on the same in-memory joins as the relay-cache HTTP API.

Preferred endpoint. Use wss://relay.bitcraftsync.app/inventory/ws for pages. Per-entity URLs (/player/<id>/inventory/ws, …/housing/ws, /claim/<id>/inventory/ws) still work but are meant for single-key debugging.

Endpoint

URLNotes
wss://relay.bitcraftsync.app/inventory/ws Public multiplexed stream (JSON text frames).

Browser WebSocket always upgrades with GET — there is no POST body on connect. Send the subscription as the first text frame after onopen.

Subscribe (client → server)

{
  "players": ["1297036692699996362"],
  "houses":  ["1297036692699996362"],
  "claims":  ["576460752317570321"],
  "player_crafts": ["1297036692699996362"],
  "claim_crafts":  ["576460752317570321"]
}
FieldMeaningHTTP equivalent
players Player entity ids → personal bags (pockets, bank, wagon, mounts, …) GET /player/<id>/inventory
houses Player entity ids → housing interiors GET /player/<id>/housing
claims Claim entity ids → shared claim storage (no Town Banks) GET /claim/<id>/inventory
player_crafts Player entity ids → progressive + passive crafts GET /player/<id>/crafts
claim_crafts Claim entity ids → crafts on claim buildings GET /claim/<id>/crafts
  • Entity ids are u64. Prefer strings in JSON (JS Number is unsafe above 253).
  • At least one non-empty list required.
  • Max 64 distinct keys total after dedupe.
  • A later subscribe frame replaces the set and re-sends snapshots.

Resolve ids via HTTP first when needed: GET /player?name=…, claim search, or claim_entity_id on player inventory bags.

Server → client frames

Initial burst, then ack

One tagged snapshot per subscribed key, then:

{ "type": "subscribed", "count": 4 }

Snapshot / live update

{
  "type": "player_inventory",
  "entity_id": "1297036692699996362",
  "data": { /* identical to the matching GET JSON */ }
}
typedata matches
player_inventoryGET /player/<id>/inventory
player_housingGET /player/<id>/housing
claim_inventoryGET /claim/<id>/inventory
player_craftsGET /player/<id>/crafts
claim_craftsGET /claim/<id>/crafts

Missing entity (still subscribed): { "type", "entity_id", "error": "player not found" } (or claim not found). After subscribed, further frames are live updates for keys that changed — full snapshots, not diffs (coalesced ~75 ms server-side).

Heartbeat (connectivity)

Every 5 seconds after subscribe completes:

{ "ts": 1753296000123 }

Unix time in milliseconds (UTC). No type field — distinguish from inventory frames by ts alone (snapshots always have type + entity_id). Treat ~15 s without a heartbeat as a dead connection and reconnect.

Errors

{ "error": "…" }

Bad JSON, empty subscribe, oversize set, etc. The socket may close afterward.

Browser example

const PLAYER = "1297036692699996362"; // from GET /player?name=…
const CLAIM = "576460752317570321";

const ws = new WebSocket("wss://relay.bitcraftsync.app/inventory/ws");
let lastBeat = Date.now();

ws.onopen = () => {
  ws.send(JSON.stringify({
    players: [PLAYER],
    houses: [PLAYER],
    claims: [CLAIM],
    player_crafts: [PLAYER],
    claim_crafts: [CLAIM],
  }));
};

ws.onmessage = (ev) => {
  const msg = JSON.parse(ev.data);
  if (msg.ts != null && msg.type == null) {
    lastBeat = Date.now(); // or msg.ts
    return;
  }
  if (msg.type === "subscribed") return; // initial burst done
  if (msg.error && !msg.data) {
    console.error("inventory ws", msg.error);
    return;
  }
  switch (msg.type) {
    case "player_inventory": /* msg.entity_id, msg.data */ break;
    case "player_housing": break;
    case "claim_inventory": break;
    case "player_crafts": break;
    case "claim_crafts": break;
  }
};

setInterval(() => {
  if (ws.readyState === WebSocket.OPEN && Date.now() - lastBeat > 15_000) {
    ws.close(); // reopen…
  }
}, 5_000);

UI semantics

  • Player inventory categories include pockets, toolbelt, wallet, bank, wagon, cache, boat, recovery, deployable (goats / birds / mounts).
  • Housing is separate (status: ok | noHouse).
  • Claim inventory is shared storage only (no Town Banks), grouped by dimension.
  • Crafts unify progressive + passive actions with recipe outputs. The WS always returns both completed and in-progress (filter client-side with completed).
  • Updates can fire with the same aggregate stack counts (internal bag churn); treat each frame as authoritative for that type+entity_id.

Ops

  • Soft cap ~512 interest leases fleet-wide; one page with ≤64 keys is fine.
  • /cache-health → streams.{active,lifetime_notifies,lifetime_pushes}.
  • Same public trust model as the HTTP cache (no auth). JSON only on the WS.
