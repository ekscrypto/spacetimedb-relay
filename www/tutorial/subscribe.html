2. Subscribe to tables

After InitialConnection, send SQL subscriptions. For a handful of small tables you can subscribe in one shot. For BitCraft-scale dumps, subscribe one query set at a time and wait for SubscribeApplied before starting the next.

Why sequencing matters. A single set-replace subscribe covering hundreds of BitCraft tables can produce a multi-hundred-MB SubscribeApplied. Some edges reset long transfers (~90 s). The relay itself uses --subscribe-chunk-size 1 upstream for the same reason. Your downstream client should mirror that discipline when pulling large working sets from the mirror.

One table

Subscribe with a single SQL query. When on_applied / SubscribeApplied fires, the matching rows are in your client cache and live TransactionUpdates will follow. 

# Inside on_connect (SDK) or after InitialConnection (raw):
QUERIES = ["SELECT * FROM chat_message_state"]

# SDK: pass QUERIES into client.run(...), or register a subscription
# builder if your SDK version exposes one. When the applied callback
# runs, iterate the local cache:
#
#   for row in client.db.chat_message_state.iter():
#       print(row)

# Raw v2 wire (conceptual): encode ClientMessage::Subscribe {
#   request_id: 1,
#   query_set_id: 1,          # new id = additive set
#   query_strings: QUERIES,
# } as BSATN, prefix is NOT needed on client→server for v2 in most
# SDK stacks (SDK adds framing). Wait for ServerMessage::SubscribeApplied
# with the same request_id before treating the table as complete.
// SDK (generated bindings)
ctx.subscription_builder()
    .on_applied(|ctx| {
        println!(
            "applied — {} chat rows",
            ctx.db.chat_message_state().count()
        );
    })
    .on_error(|_ctx, err| eprintln!("subscribe error: {err}"))
    .subscribe(["SELECT * FROM chat_message_state"]);

// Or typed query builder, if generated for that table:
// .add_query(|q| q.from.chat_message_state()).subscribe();

Many tables — synchronising strategy

Rule: do not start the next subscribe until the previous query set has applied. Apply live updates that arrive during the wait (they belong to already-applied sets). Use a fresh query_set_id for each additive set on the v2 wire; reusing an id replaces that set.

connect │ ▼ Subscribe query_set=1 [table A] │ (may receive TransactionUpdate for other activity) ▼ SubscribeApplied #1 → merge rows for A │ ▼ Subscribe query_set=2 [table B] │ ▼ SubscribeApplied #2 → merge rows for B │ ▼ … repeat … │ ▼ ready — all sets applied; stream TransactionUpdates
  1. Partition work into query sets One table per set is the safest default for huge tables (footprint_tile_state, full inventories, …). Small related tables can share a set.
  2. Send Subscribe for set i Record request_id / handle so you can match Applied.
  3. Wait for Applied (and keep applying TUs) Ignore unrelated server messages; on SubscriptionError, abort or retry that set.
  4. Only then advance i Mark ready when the last set applies. On reconnect, restart from index 0 (or your own snapshot strategy). 
import asyncio

TABLES = [
    "claim_state",
    "building_state",
    "inventory_state",
    "player_username_state",
]

async def subscribe_sequential(client):
    """Wait for each subscription's applied callback before the next."""
    for table in TABLES:
        done = asyncio.Event()

        def on_applied(_client, _msg, *, _done=done, _table=table):
            print(f"SubscribeApplied: {_table}")
            _done.set()

        # Pseudocode — wire to your SDK's subscription / callback API:
        # client.subscribe(
        #     [f"SELECT * FROM {table}"],
        #     on_applied=on_applied,
        # )
        #
        # With a raw BSATN client: send Subscribe(query_set_id=i),
        # then loop reading frames until SubscribeApplied for that id,
        # applying any TransactionUpdate seen in between.

        await done.wait()

    print("all sequential subscriptions applied")
use std::sync::{Arc, Mutex};

/// Sequential subscribe: each on_applied starts the next query set.
fn subscribe_next(ctx: &impl spacetimedb_sdk::DbContext, tables: Arc<Mutex<Vec<&'static str>>>) {
    let Some(table) = tables.lock().unwrap().pop() else {
        println!("all sequential subscriptions applied");
        return;
    };
    let tables2 = tables.clone();
    let sql = format!("SELECT * FROM {table}");
    println!("subscribing {table}");
    ctx.subscription_builder()
        .on_applied(move |ctx| {
            println!("SubscribeApplied: {table}");
            subscribe_next(&ctx, tables2);
        })
        .on_error(|_ctx, err| eprintln!("subscribe error: {err}"))
        .subscribe([sql]);
}

Wire-level notes (v2)

FieldRole
request_id Client-chosen; echoed on SubscribeApplied.
query_set_id New id appends an additive set; reuse replaces that set.
query_strings SQL list for this set, e.g. SELECT * FROM claim_state.
During wait Apply TransactionUpdates so already-subscribed sets stay live.

This is the same pattern relay-cache uses on loopback: one base multi-query set, wait for Applied, then an additive second set for hexite location_state PKs discovered from the first dump. See relay-cache joins.
